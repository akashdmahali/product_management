STEP 02 Chunking
Documents are split into smaller chunks because embedding models have token limits, and retrieval works best on focused passages. Chunk size and overlap are key hyperparameters that trade precision for context.

STEP 03 Embedding
Each chunk is converted into a dense vector — a list of numbers capturing its semantic meaning. Similar concepts produce nearby vectors in high-dimensional space. Embeddings are generated by models like text-embedding-3-small (1536-dim) or BAAI/bge-large.
🔢
Embedding Model
Converts each chunk text → a high-dimensional float vector

Click generate to visualize embeddings for each chunk.

STEP 04 Vector Store (Index)
Embeddings are stored in a vector database (e.g., Pinecone, Weaviate, Chroma, pgvector, FAISS). At query time the store performs fast Approximate Nearest Neighbour (ANN) search — cosine or dot-product similarity — over millions of vectors in milliseconds.
🗄️
Vector Index
All chunk embeddings stored and indexed for ANN search

Click to index all vectors.

STEP 05 Query and Retrieval
At inference time, the user's query is embedded with the same model, then compared to all stored vectors via cosine similarity. The top-k most similar chunks are retrieved and will be provided as context to the LLM.

STEP 06 Augmented Generation
The retrieved chunks are injected into the LLM prompt as grounding context. The model generates an answer using both its parametric knowledge and the retrieved passages — this is the "Augmented Generation" in RAG.
